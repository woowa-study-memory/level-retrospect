### 타입 안정성

<details>
<summary><b>Q. 타입안정성이란 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 잘못된 값을 사용하는 것을 컴파일 단계에서 막는 것이다.
  - 기본 타입 시스템이 컴파일 단계에서 오류를 막는 것으로 알고 있다.
- **결론 :**
  - 타입 안정성이란, 잘못된 타입 사용을 컴파일 시점에 방지하여 프로그램의 오류를 줄이는 성질이다.

</div>
</details>

<details>
<summary><b>Q. 컴파일 타입 체크와 런타임 타입 체크의 차이는 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 컴파일 타입 체크는 컴파일러 실행 시 체크하는 오류를 의미한다 런타입 타입 체크는 프로그램 실행 중 진행하는 타입 체크이다.
  - 즉, 런타임은 프로그램 실행 중에 진행하므로 오류 발생 시 개발자가 예외(classCastException)를 처리해야하며 오류 발생 시 프로그램이 멈출 수 있는 위험성이 존재한다.
- **의문점 :**
  - 런타임 시점 타입 체크는 어떤 것으로 하는가? 
- **배움 후 수정된 생각 :**
  - 개발자로서 런타임 시점 타입 체크는 최소화해야한다. 인스턴스오브를 사용하는 대신 객체를 분리해 책임을 맡기면 더 좋은 프로그래밍이 되지 않을까?
  - 런타임 시점 타입 체크는 반드시 필요한 상황이 나올 것이다. 이럴 때는 예외에 대한 처리를 철저히 해야한다.
  - `instanseof`, 명시적 캐스팅
- **결론 :**
  - 컴파일 타입 체크와 런타임 타입 체크의 차이점은 오류 체크 시점이 다르다. 컴파일 타입 체크는 컴파일러 실행 시 오류를 체크하며 런타임 타입 체크는 프로그램 실행 중에 진행한다.

</div>
</details>

<details>
<summary><b>Q. 제네릭은 타입 안정성에 무엇을 기여하는가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 제네릭은 타입 안정성 측면에서 `Object`보다 세분화된 타입을 제공한다.
  - 이를 통해 컴파일 시점에서 타입 안정성을 보장한다.
  - 또한 개발자에게 명시적으로 어떠한 종류의 변수가 들어오는 지를 보여줄 수 있으며 이를 이용해 재사용성까지 챙길 수 있다.
- **결론 :**
  - 제네릭은 컴파일 타임 타입 체크를 통해 타입 안정성을 보장하고, 코드의 명확성과 재사용성을 높이는 자바의 핵심 기능이다.

</div>
</details>

  - **어떻게 사용하는가?**
    - **데이터와 로직의 응집 (상태와 행위 결합):** 단순 상수 나열을 넘어, 상수에 연관된 데이터(예: 에러 코드 번호, 한글 설명)와 로직을 `enum` 내부에 캡슐화하여 객체지향적으로 사용한다.

### Optional

<details>
<summary><b>Q. Optional이 null보다 좋은 점은 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - `Optional`은 완벽하게 null을 방지하는 것은 아니지만 안전한 `null` 처리를 위한 도구이다.
  - 즉, 명시적으로 개발자에게 해당 코드에서 `null` 발생 가능성을 보여주며 더 철저한 `null` 처리를 진행하게 한다.
- **결론 :**
  - `Optional`은 개발자에게 `null` 가능성을 경고한다. 그렇기에 철저하고 안전한 `null` 처리를 유도한다.
  - 그러나 항상 `Optional`이 좋은 것은 아니다. `Optional`도 일종의 래퍼 클래스이기에 객체 생성 비용과 메모리 오버헤드가 존재한다.
  - 또한 가독성 저하 가능성도 존재하기에 적절한 학습과 사용이 필요하다. 

</div>
</details>

<details>
<summary><b>Q. Optional이 null-safety를 보장해주는가? </b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 아니다. `Optional`은 `null-safety`를 보장하지 않는다. 그 대신 개발자에게 명시적인 의도 전달을 하며 `null` 처리를 강제하여 보다 안전하게 한다.
- **결론 :**
  - **도구일 뿐, 시스템적 보장은 아니다:** `Optional` 객체 변수 자체에 `null`을 할당하거나, 내부가 비어있는데 확인 없이 무작정 `.get()`을 호출하면 여전히 예외(NPE 또는 `NoSuchElementException`)가 발생한다. 코틀린처럼 컴파일 단에서 null을 완벽히 차단하는 진짜 'null-safety'와는 다르다.
  - **API 설계와 가독성의 무기:** 메서드의 반환 타입으로 사용됨으로써, 코드를 호출하는 사람에게 "이 결과는 비어있을 수 있으니 대비하라"는 강력한 시각적/구조적 경고를 보낸다.
  - **올바른 방어 코드의 유도:** 개발자가 낡은 방식인 `if (obj == null)` 대신, `.orElse()`, `.orElseThrow()`, `.ifPresent()` 등의 함수형 메서드를 사용하도록 유도하여 더 우아하게 예외 처리 흐름을 만들도록 돕는다.

</div>
</details>

<summary><b>Q. Null Object Pattern</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - `Null Object Pattern`은 `null`을 다루는 다른 방식이다.
  - 간단하게 `Optional`이 명시적인 `null` 처리를 유도했다면 `Empty` 상태를 하나의 객체로 모델링하는 패턴이다.
- **의문점 :**
  -  단점은 없는가? 
- [배움 후 수정된 생각 :](https://github.com/quddaz/woowa-study/blob/main/%EB%94%94%EC%9E%90%EC%9D%B8%20%ED%8C%A8%ED%84%B4/Null%20Object%20Pattern.md)
  - 장점
    - `null-safety`구현 가능
    - `null` 분기 문 삭제
    - 다형성 활용
  - 단점
    - 버그 은닉 : `Empty` 객체에서 모든 에러를 잡아먹을 수 있다.
    - 의미 왜곡 : `Empty`를 모델링하기에 도메인적으로 정의 가능한지 여부가 먼저 제대로 정의해야한다.
    - 과도한 추상화
- **결론 :**
  - (Null Object Pattern은 단순히 null을 제거하는 기법이 아니라,
“비어있음(Empty)를 하나의 도메인 객체로 모델링하는 방식”이다.
  - 따라서 다음 기준을 만족할 때 사용하는 것이 적절하다:
    - “없음”이 정상 상태이며 도메인적으로 의미가 있을 때
    - 해당 객체가 없어도 로직이 자연스럽게 계속 흘러가야 할 때
    - if-null 분기가 반복되며, 이를 다형성으로 치환할 수 있을 때
    - Empty 객체의 모든 메서드가 도메인적으로 자연스럽게 정의 가능할 때

</div>
</details>

<summary><b>Q. State Pattern</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - `state Pattern`은 객체 내부 상태가 변경될 때 객체의 행동이 달라지도록 하는 행동 패턴이다.
  - 이 패턴을 사용하면 객체는 마치 클래스가 변경된 것처럼 서로 다른 행동을 수행하는 것처럼 보이게 한다.
  - 장점 :
    - **조건 분기 문 취소화** : 조건을 통해 동작을 분기하는 코드를 최소화하여 각 객체가 맡는 형태로 진행할 수 있다.
  - 단점 :
    - **클래스 폭발** : 모든 상태에 대한 정의 및 다형성을 사용하기에 클래스가 매우 많이 늘어납니다.
- **의문점 :**
  - 어느 경우에 상태 패턴이 적절할까?
- [배움 후 수정된 생각](https://github.com/quddaz/woowa-study/blob/main/%EB%94%94%EC%9E%90%EC%9D%B8%20%ED%8C%A8%ED%84%B4/State%20Pattern.md)
  - **도입을 고민해야하는 시점**
    - 상태의 종류가 많고 유동적인 경우
    - 상태 전이 규칙이 자주 변경되는 경우
    - 상태 별 행동이 복잡하거나 상이한 경우 
- **결론 :**
  - `State Pattern`은 단순히 `if-else`를 제거하기 위한 패턴이 아니라,
**“상태에 따라 달라지는 행동을 객체로 분리하여 관리하기 위한 설계 방식”**이다.
  - 따라서 다음과 같은 경우에 사용하는 것이 적절하다:
    - 객체의 상태에 따라 행동이 크게 달라질 때
    - 조건 분기(if/switch)가 반복되며 점점 복잡해질 때
    - 상태 전이 규칙과 상태별 책임을 명확히 분리할 필요가 있을 때
    - 상태 변경 및 확장이 빈번하여 OCP(개방-폐쇄 원칙)를 지키고 싶을 때
  - 반대로 아래와 같은 경우에는 사용을 지양한다:
    - 상태 종류가 적고 변경 가능성이 거의 없는 경우
    - 단순 분기(if 한두 개)로 충분히 해결 가능한 경우
    - 상태 객체로 분리했을 때 오히려 이해 비용이 증가하는 경우


</div>
</details>

  - 정적 팩토리의 장점은 무엇인가?
     - 생성 목적에 대해 표현이 가능하다.
     - 반환 타입을 은닉할 수 있다.(추상 클래스에 의존)
- **결론 :**
  - 정적 팩토리 메소드 패턴은 정적 메소드를 통해 간접적으로 생성자를 호출하는 객체 생성 패턴이다.
  - **어떤 상황에서 사용해야하는가?**
      - 생성에 `이름(의도)`가 필요할 때
      - 생성 방식이 여러 개일 때
      - 생성 로직이 단순하지 않고 가독성이 필요할 때
      - 구현체를 숨기고 싶을 때

</div>
</details>

<summary><b>Q. Strategy Pattern</b></summary>
<div markdown="1">

<br>

[전략패턴](https://github.com/quddaz/woowa-study/blob/main/%EB%94%94%EC%9E%90%EC%9D%B8%20%ED%8C%A8%ED%84%B4/Strategy%20Pattern.md)

- **내 생각 :**
  - 전략 패턴은 어떤 일을 수행하는 방법이 여러 가지일 때, 합성의 원칙을 이용해서 실행 중에 이리러지 교체할 수 있게 만드는 패턴입니다.
  - 합성의 장점처럼 인터페이스에 의존하며 유연성과 확장성에 강합니다. 
- **결론 :**
  - **정의:** 로직을 각각의 클래스로 캡슐화하여, 클라이언트 코드의 변경 없이 런타임에 동적으로 전략을 교체할 수 있게 해주는 디자인 패턴.
  - **핵심 원칙:** 상속이 아닌 **합성**을 활용한다. 행위를 직접 물려받는 것이 아니라, 행위를 수행할 부품(인터페이스)을 외부에 뚫어둔 구멍에 주입받아 조립한다.
  - **OCP (개방-폐쇄 원칙) 준수:** 기존 컨텍스트(캐릭터, 장바구니 등) 코드는 단 한 줄도 수정할 필요 없이, 새로운 전략 클래스만 추가하면 무한하게 기능을 확장할 수 있다.

</div>
</details>

<summary><b>Q. Model View Controller Pattern</b></summary>
<div markdown="1">

<br>

[MVC 패턴](https://github.com/quddaz/woowa-study/blob/main/%EB%94%94%EC%9E%90%EC%9D%B8%20%ED%8C%A8%ED%84%B4/Model%20View%20Controller%20Pattern.md)

- **내 생각 :**
  - MVC 패턴은 거대한 애플리케이션의 뼈대를 잡는 아키텍처 패턴이다.
  - Model/View/Controller 를 각각 분리하여 자신만의 역할을 하게하는 것이 특징입니다.
  - 계층화가 중요하지만 그만큼 역할에 맞는 의존성 관리 또한 필수입니다.
  - 이를 통해 변경 및 확장에 대해 역할에 맞는 사이드 이펙트를 갖게하는 특징이 있습니다. 
- **결론 :**
  - MVC 패턴은 단순히 역할을 나누는 것이 아니라, **“변경 이유가 다른 것들을 분리하여 유지보수성과 확장성을 높이기 위한 구조”**이다.
