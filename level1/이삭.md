### 캡슐화

<details>
<summary><b>Q. 캡슐화란 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 외부에서 객체 내부의 세부사항에 직접 접근하거나 알지 못하도록 보호하는 것
  - 외부에서 getter를 사용하여 객체의 상태를 직접 꺼내 사용하지 않는 것
    
- **의문점 :**
  - 접근 제어자를 private으로 하면 캡슐화가 되는 것인가?
    
- **배움 후 수정된 생각 :**
  - 캡슐화는 단순히 **접근을 막는 것(private)** 이 아니라,
    객체의 상태와 행동을 하나로 묶고, 외부에는 **'의도된 인터페이스'** 만 노출하는 것이다.
    
  - private은 캡슐화를 도와주는 수단일 뿐, 그 자체가 캡슐화는 아니다.
    
    -> private인데도 getter로 상태를 다 꺼내 쓰면 사실상 캡슐화가 깨진 상태다.
    
  - 중요한 것은
    1. “외부에서 상태를 직접 다루게 할 것인가?”
    2. “객체에게 일을 시킬 것인가?”

    둘 중에 어느 것을 선택하냐 이다.
    
  - 캡슐화가 잘 된 객체는
    - 자신의 상태를 스스로 변경하고,
    - 외부는 "무엇을 할지"만 요청한다 (how가 아니라 what)
    - 상태를 꺼내서 판단하는 게 아니라, 객체에게 책임을 위임한다.
      
- **결론 :**
  - 캡슐화가 되었는지 판단 기준:
    1. 외부에서 객체의 상태를 직접 다루고 있지 않은가?
    2. 객체가 자신의 상태 변경 책임을 스스로 가지고 있는가?
    3. 외부는 '무엇을 해라'만 요청하고, '어떻게'는 객체가 결정하는가?
       
  - 따라서
    - getter는 "무조건 나쁜 것"은 아니지만
    - 비즈니스 로직에서 상태를 꺼내 쓰기 시작하면 캡슐화가 깨진다고 본다.
      
  - 실제 코드 작성 시에는
    - 상태를 꺼내기 전에 "이걸 객체에게 시킬 수 없을까?"를 먼저 고민한다.
    - 로직이 외부에 퍼지기 시작하면 해당 로직을 객체 내부로 이동시킨다.

</div>
</details>

### 추상화

<details>
<summary><b>Q. 추상화란 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 사물이나 개념에서 공통되는 본질을 추출하여 단순화하는 것
  - 객체지향 프로그래밍적으로는 객체의 공통 속성이나 메서드를 추출하여 상위클래스에 정의하는 것
  - 상위클래스에서 메서드를 구현하지 않고 시그니처만 남긴 후 하위클래스에서 구현하는 것
    
- **의문점 :**
  - 공통적인 부분이 있으면 추상화하라는 신호로 보면 되는 것인가?
    
- **배움 후 수정된 생각 :**
  - 추상화는 단순히 공통 속성이나 메서드를 상위 클래스에 올리는 것이 아니라,
    불필요한 세부사항을 숨기고 핵심적인 역할만 드러내는 것
    
  - 따라서 추상화는 상속뿐만 아니라
    인터페이스를 통한 역할 정의, 객체 간 협력 구조 설계 등 다양한 방식으로 표현될 수 있다.
    
  - 공통적인 부분이 보인다고 해서 무조건 추상화를 적용하는 것은 적절하지 않다.
    -> **"함께 변경되는가?"**, **"같은 책임을 가지는가?"** 를 기준으로 판단해야 한다.
    
  - 특히 여러 객체가 동일한 책임을 수행하지만 내부 구현이 다르다면,
    이는 같은 "종류"라기보다 같은 "역할"을 수행하는 것이므로
    인터페이스를 통해 역할 중심으로 추상화하는 것이 더 적절하다.
    
  - 반대로, 객체들이 구조와 상태까지 유사하며
    "A는 B다"라는 관계(is-a)가 자연스럽고 공통 로직을 공유해야 한다면
    그때는 상속을 통한 추상화가 적절하다.
  
- **결론 :**
  - 추상화를 적용할 때 다음 기준으로 판단한다:
    1. 이 코드들이 같은 책임을 가지는가?
    2. 함께 변경되는가? -> 같은 이유로 수정 되는가?
    3. 공통점이 단순한 '기능'인지, 구조까지 포함된 '종류'인지 판단한다
       
  - 위 기준을 바탕으로,
    - 같은 역할만 공유한다면 → 인터페이스로 추상화
    - 같은 종류이며 상태와 로직을 함께 공유한다면 → 상속으로 추상화
      
  - 즉, 추상화는
    
    👉 공통점을 단순히 묶는 것이 아니라
    
    👉 책임과 역할을 기준으로 구조를 나누고, 변화에 유연한 코드를 만들기 위한 설계 방법이다.
</div>
</details>

### 결합도
<details>
<summary><b>Q. 결합도란 무엇인가? </b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 한 객체가 다른 객체를 알고 있는 정도
    
- **의문점 :**
  - 의존과는 무슨 차이인가?
    
- **배움 후 수정된 생각 :**

  - 결합도는 객체 간의 **의존 관계가 얼마나 강하게 묶여 있는지를 나타내는 정도**이다.
    
  - 의존과 결합도의 차이는 다음과 같다:
    - **의존(Dependency)**  

      -> 한 객체가 다른 객체를 사용하는 **관계** 자체

    - **결합도(Coupling)**  

      -> 그 의존 관계가 **얼마나 강하게 연결되어 있는지 (변경이 얼마나 전파되는지)**

  - 쉽게 비유하면:

    - 의존: "나는 너가 필요해"
    - 결합도: "근데 너랑 너무 붙어 있어서 너 바뀌면 나도 같이 바뀌어"

  - 예를 들어:

    - 블루투스 이어폰 -> 여러 기기에서 사용 가능 -> **결합도 낮음**
    - 특정 기기 전용 이어폰 -> 다른 기기에서는 사용 불가 -> **결합도 높음**

  - 코드에서의 차이:

    - 객체를 직접 생성하고 특정 구현에 의존하면 -> **결합도 높음**
    - 인터페이스에 의존하고 외부에서 주입받으면 -> **결합도 낮음**

  - 중요한 점:

    - 의존은 프로그램에서 **필수적으로 존재**한다.
    - 하지만 결합도는 **설계를 통해 낮출 수 있다.**

- **결론 :**

  - 결합도의 판단기준:
    1. 이 객체가 변경되었을 때 다른 객체도 함께 수정되는가?
    2. 특정 구현에 강하게 묶여 있어서 교체하기 어려운가?
    3. 객체를 쉽게 다른 것으로 바꿔 끼울 수 있는가?

  - 설계할 때는
    - 의존 자체를 없애려고 하지 않고,
    - **결합도를 낮추는 방향으로 설계한다.**

  - 이를 위해
    - 추상(인터페이스)에 의존하고,
    - 객체 생성은 외부에서 주입받고,
    - 상태를 꺼내기보다 객체에게 메시지를 보내도록 한다.

</div>
</details>
