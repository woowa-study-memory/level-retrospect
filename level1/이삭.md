### 객체지향 기초 개념

<details>
<summary><b>Q. 객체란 무엇이며, 객체와 클래스의 차이는 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 객체: 클래스를 통해 실제로 만들어진 결과물
  - 클래스: 객체를 만들기 위한 설계도
    
- **의문점 :**
  - 인스턴스는 무엇인가?
    
- **배움 후 수정된 생각 :**

  - 객체는 **상태와 행동을 가지며 다른 객체와 메시지를 주고받으며 상호작용(협력)하는 주체**이다.
    
  - 인스턴스는 "특정 클래스로부터 생성된 객체"를 의미하며,  

    &rarr; 객체와 거의 같은 의미로 사용되지만  

    &rarr; **"어떤 클래스로부터 만들어졌는지"를 강조하는 표현**이다.

  - 즉,
    - 객체: 실행 시점에 존재하며 협력의 주체
    - 클래스: 객체를 생성하기 위한 정의
    - 인스턴스: 특정 클래스에 속한 객체

- **결론 :**

  - 객체를 단순히 "클래스로부터 만들어진 결과물"로 보지 않고  

    &rarr; **자신의 상태와 행동을 가지고, 책임을 수행하며, 다른 객체와 협력하는 주체**로 이해한다.

  - 클래스는 객체를 만들기 위한 수단일 뿐이며,  

    &rarr; 설계 시에는 클래스보다 **객체의 책임과 협력 관계를 먼저 고민한다.**

  따라서 객체지향 설계에서는 어떤 클래스를 만들지보다 **어떤 객체가 어떤 책임을 가지고 어떻게 협력할지**를 먼저 정의하는 것이 중요하다.

---

</div>
</details>

<details>
<summary><b>Q. VO와 도메인 객체의 차이는 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - VO는 원시값을 의미있는 타입으로 감싼 객체
  - 도메인 객체는 협력에 참여하는 객체
  - 따라서 둘의 차이점은 비즈니스로직에서 협력에 참여하냐 안하냐의 차이점일 것 같다.
    
- **의문점 :**
  - VO와 도메인 객체가 무엇인지, 둘의 차이가 무엇인지에 대한 나의 답변이 올바른가?
    
- **배움 후 수정된 생각 :**
  - 도메인은 내가 개발하려는 영역 
  - 도메인 객체는 "도메인을 표현하는 모든 객체"를 의미하는 상위 개념이며, Entity와 VO를 모두 포함한다.
  - VO는 도메인 객체의 한 종류로, **값 자체가 의미이며 값으로 동일성을 판단하는 불변 객체**이다.
  - 따라서 VO와 도메인 객체는 서로 비교되는 개념이 아니라  

    &rarr; **"포함 관계 (VO in 도메인 객체)"** 이다.

  - 또한 VO 역시 도메인 객체이기 때문에 협력에 참여하며,  

    &rarr; **"협력 여부"로 둘을 구분하는 것은 잘못된 기준이다.**

  - 올바른 구분 기준은 다음과 같다:
    - Entity: 식별자를 통해 동일성을 판단 (값이 변해도 같은 객체)
    - VO: 값 자체로 동일성을 판단 (값이 같으면 같은 객체)
    
- **결론 :**
  - VO는 도메인 객체의 한 종류이므로, 둘은 비교 대상이 아니라 포함 관계이다.
  - 설계 시에는 "도메인 객체인가?"를 먼저 판단하기보다  

    &rarr; 해당 객체가 **값 중심인지(VO)**, **존재 중심인지(Entity)** 를 기준으로 구분한다.

  - 따라서 객체를 설계할 때는
    - 값이 동일하고 같은 객체로 봐야 한다면 VO로 만들고,
    - 상태가 변해도 동일한 존재로 유지되어야 한다면 Entity로 설계한다.
      
  - VO는 값이 같으면 같다고 판단해야하므로 eqauls()와 hashCode()가 필수적이다.
  
</div>

<details>
<summary><b>Q. 일급 컬렉션을 왜 사용하는가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 일급컬렉션: 내부에 상태를 컬렉션 하나만 가진 객체
  - 컬렉션 자료구조 자체를 쓰면 검증로직이나 비즈니스로직을 여러군데 분산되어 응집도가 떨어지는데,
  - 일급컬렉션으로 묶는다면 한 곳에 검증로직이나 비즈니스로직을 모을 수가 있어 응집도를 높여주기때문에 사용한다.
  - 또한, 더 의미있게 표현할 수 있어 가독성이 향상된다.
    
- **의문점 :**
  - 일급컬렉션에 대해 알고있는 나의 생각이 올바른가?
    
- **배움 후 수정된 생각 :**
  - 일급 컬렉션은 단순히 컬렉션을 감싼 객체가 아니라,  

    &rarr; **컬렉션과 관련된 도메인 개념과 규칙을 함께 표현하는 객체**이다.

  - 컬렉션을 직접 사용하는 경우:
    - 검증 로직이나 비즈니스 로직이 여러 곳에 분산되고,
    - 동일한 규칙이 중복되며,
    - 변경 시 수정 범위가 넓어진다.

  - 일급 컬렉션을 사용하는 경우:
    - 컬렉션과 관련된 로직을 한 곳에 모을 수 있어 응집도가 높아지고
    - 객체 스스로 책임을 가지게 되어 캡슐화가 강화된다.

  - 또한 단순한 List, Map 등등 이 아닌,

    &rarr; "의미 있는 개념(예: Hand, Board, Participants)으로 표현되기 때문에 코드의 의도를 더 명확하게 드러낼 수 있다.

  즉, 일급 컬렉션은 단순한 자료구조를 넘어서 **도메인 객체(주로 VO)로서 역할을 한다.**

- **결론 :**

  - 일급 컬렉션 사용 기준:
    1. 컬렉션에 대한 **도메인 규칙(검증, 계산 등)** 이 존재할 때
    2. 해당 컬렉션이 **하나의 의미 있는 개념으로 표현될 수 있을 때**
    3. 동일한 로직이 여러 곳에서 반복되고 있을 때

  - 단순히 데이터를 담기 위한 컬렉션이라면 사용하지 않고,  

    &rarr; **도메인 로직과 책임을 부여할 필요가 있을 때만 사용한다.**

  - 따라서 일급 컬렉션은
    - 응집도를 높이고,
    - 중복을 제거하며,
    - 도메인 표현력을 강화하기 위한 설계 도구이다.

</div>
</details>

### 캡슐화

<details>
<summary><b>Q. 캡슐화란 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 외부에서 객체 내부의 세부사항에 직접 접근하거나 알지 못하도록 보호하는 것
  - 외부에서 getter를 사용하여 객체의 상태를 직접 꺼내 사용하지 않는 것
    
- **의문점 :**
  - 접근 제어자를 private으로 하면 캡슐화가 되는 것인가?
    
- **배움 후 수정된 생각 :**
  - 캡슐화는 단순히 **접근을 막는 것(private)** 이 아니라,
    객체의 상태와 행동을 하나로 묶고, 외부에는 **'의도된 인터페이스'** 만 노출하는 것이다.
    
  - private은 캡슐화를 도와주는 수단일 뿐, 그 자체가 캡슐화는 아니다.
    
    -> private인데도 getter로 상태를 다 꺼내 쓰면 사실상 캡슐화가 깨진 상태다.
    
  - 중요한 것은
    1. “외부에서 상태를 직접 다루게 할 것인가?”
    2. “객체에게 일을 시킬 것인가?”

    둘 중에 어느 것을 선택하냐 이다.
    
  - 캡슐화가 잘 된 객체는
    - 자신의 상태를 스스로 변경하고,
    - 외부는 "무엇을 할지"만 요청한다 (how가 아니라 what)
    - 상태를 꺼내서 판단하는 게 아니라, 객체에게 책임을 위임한다.
      
- **결론 :**
  - 캡슐화가 되었는지 판단 기준:
    1. 외부에서 객체의 상태를 직접 다루고 있지 않은가?
    2. 객체가 자신의 상태 변경 책임을 스스로 가지고 있는가?
    3. 외부는 '무엇을 해라'만 요청하고, '어떻게'는 객체가 결정하는가?
       
  - 따라서
    - getter는 "무조건 나쁜 것"은 아니지만
    - 비즈니스 로직에서 상태를 꺼내 쓰기 시작하면 캡슐화가 깨진다고 본다.
      
  - 실제 코드 작성 시에는
    - 상태를 꺼내기 전에 "이걸 객체에게 시킬 수 없을까?"를 먼저 고민한다.
    - 로직이 외부에 퍼지기 시작하면 해당 로직을 객체 내부로 이동시킨다.

</div>
</details>

### 추상화

<details>
<summary><b>Q. 추상화란 무엇인가?</b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 사물이나 개념에서 공통되는 본질을 추출하여 단순화하는 것
  - 객체지향 프로그래밍적으로는 객체의 공통 속성이나 메서드를 추출하여 상위클래스에 정의하는 것
  - 상위클래스에서 메서드를 구현하지 않고 시그니처만 남긴 후 하위클래스에서 구현하는 것
    
- **의문점 :**
  - 공통적인 부분이 있으면 추상화하라는 신호로 보면 되는 것인가?
    
- **배움 후 수정된 생각 :**
  - 추상화는 단순히 공통 속성이나 메서드를 상위 클래스에 올리는 것이 아니라,
    불필요한 세부사항을 숨기고 핵심적인 역할만 드러내는 것
    
  - 따라서 추상화는 상속뿐만 아니라
    인터페이스를 통한 역할 정의, 객체 간 협력 구조 설계 등 다양한 방식으로 표현될 수 있다.
    
  - 공통적인 부분이 보인다고 해서 무조건 추상화를 적용하는 것은 적절하지 않다.
    -> **"함께 변경되는가?"**, **"같은 책임을 가지는가?"** 를 기준으로 판단해야 한다.
    
  - 특히 여러 객체가 동일한 책임을 수행하지만 내부 구현이 다르다면,
    이는 같은 "종류"라기보다 같은 "역할"을 수행하는 것이므로
    인터페이스를 통해 역할 중심으로 추상화하는 것이 더 적절하다.
    
  - 반대로, 객체들이 구조와 상태까지 유사하며
    "A는 B다"라는 관계(is-a)가 자연스럽고 공통 로직을 공유해야 한다면
    그때는 상속을 통한 추상화가 적절하다.
  
- **결론 :**
  - 추상화를 적용할 때 다음 기준으로 판단한다:
    1. 이 코드들이 같은 책임을 가지는가?
    2. 함께 변경되는가? -> 같은 이유로 수정 되는가?
    3. 공통점이 단순한 '기능'인지, 구조까지 포함된 '종류'인지 판단한다
       
  - 위 기준을 바탕으로,
    - 같은 역할만 공유한다면 → 인터페이스로 추상화
    - 같은 종류이며 상태와 로직을 함께 공유한다면 → 상속으로 추상화
      
  - 즉, 추상화는
    
    👉 공통점을 단순히 묶는 것이 아니라
    
    👉 책임과 역할을 기준으로 구조를 나누고, 변화에 유연한 코드를 만들기 위한 설계 방법이다.
</div>
</details>

### 결합도와 응집도
<details>
<summary><b>Q. 결합도란 무엇인가? </b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 한 객체가 다른 객체를 알고 있는 정도
    
- **의문점 :**
  - 의존과는 무슨 차이인가?
    
- **배움 후 수정된 생각 :**

  - 결합도는 객체 간의 **의존 관계가 얼마나 강하게 묶여 있는지를 나타내는 정도**이다.
    
  - 의존과 결합도의 차이는 다음과 같다:
    - **의존(Dependency)**  

      -> 한 객체가 다른 객체를 사용하는 **관계** 자체

    - **결합도(Coupling)**  

      -> 그 의존 관계가 **얼마나 강하게 연결되어 있는지 (변경이 얼마나 전파되는지)**

  - 쉽게 비유하면:

    - 의존: "나는 너가 필요해"
    - 결합도: "근데 너랑 너무 붙어 있어서 너 바뀌면 나도 같이 바뀌어"

  - 예를 들어:

    - 블루투스 이어폰 -> 여러 기기에서 사용 가능 -> **결합도 낮음**
    - 특정 기기 전용 이어폰 -> 다른 기기에서는 사용 불가 -> **결합도 높음**

  - 코드에서의 차이:

    - 객체를 직접 생성하고 특정 구현에 의존하면 -> **결합도 높음**
    - 인터페이스에 의존하고 외부에서 주입받으면 -> **결합도 낮음**

  - 중요한 점:

    - 의존은 프로그램에서 **필수적으로 존재**한다.
    - 하지만 결합도는 **설계를 통해 낮출 수 있다.**

- **결론 :**

  - 결합도의 판단기준:
    1. 이 객체가 변경되었을 때 다른 객체도 함께 수정되는가?
    2. 특정 구현에 강하게 묶여 있어서 교체하기 어려운가?
    3. 객체를 쉽게 다른 것으로 바꿔 끼울 수 있는가?

  - 설계할 때는
    - 의존 자체를 없애려고 하지 않고,
    - **결합도를 낮추는 방향으로 설계한다.**

  - 이를 위해
    - 추상(인터페이스)에 의존하고,
    - 객체 생성은 외부에서 주입받고,
    - 상태를 꺼내기보다 객체에게 메시지를 보내도록 한다.

</div>

<summary><b>Q. 응집도란 무엇인가? </b></summary>
<div markdown="1">

<br>

- **내 생각 :**
  - 같은 이유로 변경되어야하는 부분이 여러군데 퍼져있는 것이 아닌, 한 곳에 모여있는 정도
    
- **의문점 :**
  - 변경의 단위는 무엇인가?
    
- **배움 후 수정된 생각 :**
  - 응집도는 단순히 코드가 모여있는 정도가 아니라, **하나의 객체가 하나의 책임에 얼마나 집중하고 있는지를 나타내는 정도**이다.

  - 여기서 중요한 기준은 "같은 이유로 변경되는가?" 인데, 이때의 **변경 이유는 데이터(필드) 단위가 아니라 "책임 단위"로 판단해야 한다.**

  - 예를 들어:

    ```java

    class User {
        void changeName(String name) { ... }
        void changePassword(String password) { ... }
    }

    ```

    - 이름 정책이 바뀌거나 비밀번호 정책이 바뀌더라도 이는 각각 다른 변경이 아니라  

      -> **"사용자 정보 관리 정책이 변경된다"는 하나의 이유로 묶인다.**

    - 따라서 위 클래스는 하나의 책임(사용자 정보 관리)에 집중하고 있어  

      -> **응집도가 높은 상태이다.**

  
  - 반면 아래와 같은 경우:

    ```java

    class User {
        void changeName(String name) { ... }
        void sendEmail() { ... }
        void calculateDiscount() { ... }
    }

    ```

    - 이름 변경 -> 사용자 정보 정책 변경
    - 이메일 발송 -> 외부 통신 방식 변경
    - 할인 계산 -> 비즈니스 정책 변경

    -> 서로 다른 이유로 변경되므로 **응집도가 낮은 상태이다.** 

  - 즉, 응집도를 판단할 때는  

    -> "이 코드들이 같은 데이터를 다루는가?"가 아니라  

    -> **"같은 책임 안에 있는가?"를 기준으로 본다.**

- **결론 :**

  - 응집도 판단 기준:
    1. 이 클래스는 하나의 책임으로 설명할 수 있는가?
    2. 변경 이유를 하나로 묶어서 설명할 수 있는가?
    3. 이 기능들을 같은 객체가 담당하는 것이 자연스러운가?

  - 설계할 때는
    - 필드 기준이 아니라 **책임** 기준으로 묶고
    - 서로 다른 이유로 변경되는 코드는 분리한다.

  응집도를 높이기 위해 무조건 잘게 나누는 것이 아니라, **책임이 분리되는 지점까지만 나누는 것이 중요하다.**

</div>
